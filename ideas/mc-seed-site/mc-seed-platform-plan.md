## 1. MC 种子机制技术说明

### 1.1 种子的本质

Minecraft 的种子（Seed）是一个 64 位整数（Java 版）或 32 位整数（基岩版），作为伪随机数生成器（PRNG）的初始值，控制地形噪声、群系分布、结构放置等世界生成的全部过程。相同版本 + 相同端 + 相同种子 → 必然生成相同世界，这是确定性保证。

### 1.2 原版客户端 vs 服务端：完全一致

官方单人客户端和原版服务端使用相同的世界生成代码，同一个种子生成的世界逐块相同，无需区分。

### 1.3 Mod 环境（NeoForge / Forge）：分情况处理

| Mod 类型 | 对种子世界生成的影响 |
|---|---|
| 纯内容 mod（新物品、新生物、新机制，不改地形） | 不影响，地形与原版一致 |
| 添加新群系 / 新结构的 mod（Terralith、Oh The Biomes 等） | **完全破坏种子一致性**，世界面目全非 |
| 只改矿石 / 装饰生成的 mod | 地形骨架相同，细节（矿脉、植被）随机化 |

**已知问题**：NeoForge 本体在某些版本中存在生物群系生成差异 bug（如同一种子在原版生成丛林，NeoForge 却生成平原），这是框架层面的问题，与具体 mod 无关。

**平台设计结论**：

- 默认收录 **纯原版种子**，这是主体。
- Mod 整合包种子单独建立分类，**必须注明整合包名称 + 精确版本号**，不可与原版种子混用。
- 投稿表单中"Mod 环境"字段应设为必填且有明确提示。

### 1.4 Java 版 vs 基岩版：地形接近但结构不同

自 1.18 起 Mojang 做了大规模跨版本种子同步：

- **地形、群系布局**：相同种子在两端大体一致（前提：种子数值 < 4,294,967,296）
- **结构位置**（村庄、要塞、神庙等）：**仍然不同**，不可互用
- **山地、洞穴、矿石生成**：存在细节差异
- **种子空间**：Java 版 2⁶⁴，基岩版 2³²；Java 大数种子在基岩版无法使用

**结论**：Java 版和基岩版种子必须作为两个独立字段，不允许跨版本标注"通用"（除非经过实测验证且数值在安全范围内）。

## 2. 版本号差异分析

### 2.1 三层精度模型

版本兼容性分析需要区分三个层次，对种子有效性的影响截然不同：

#### 第一层：大版本跨越 → 种子往往完全失效

| 版本变化 | 核心变化 | 种子兼容性 |
|---|---|---|
| 1.17 → 1.18 | 世界高度从 256 扩展到 320，地形算法彻底重构，洞穴系统完全重做 | ❌ 完全不兼容 |
| 1.12 → 1.13 | 世界生成架构重写 | ❌ 完全不兼容 |
| 1.7 → 1.8 | 群系生成算法变更 | ❌ 完全不兼容 |

这类跨越后，同一种子生成的是完全不同的世界。

#### 第二层："游戏更新包"（Game Drop）内的小版本跨越 → 部分失效

Mojang 自 1.21 起将每个小版本作为独立更新包发布，每次都可能带来世界生成变化。**1.21 系列关键断层点**：

| 版本 | 更新包名称 | 关键世界生成变化 | 与前版兼容性 |
|---|---|---|---|
| 1.21.0 | Tricky Trials | 引入试炼室（Trial Chamber） | — |
| 1.21.1 | 补丁版本 | 无世界生成变化 | ✅ 与 1.21.0 兼容 |
| 1.21.2 | Bundles of Bravery | 苍白花园（Pale Garden）以实验性内容首次出现 | ⚠️ 断层开始 |
| **1.21.4** | **The Garden Awakens** | **苍白花园正式引入，成为独立地上群系** | ❌ 与 1.21.1 不兼容（苍白花园位置、大小全部不同） |
| **1.21.5** | Spring to Life | 进一步调整世界生成，苍白花园分布权重变化 | ⚠️ 与 1.21.4 存在可见差异 |

**实际意义举例**：一颗在 1.21.1 中出生点旁边有苍白花园的"卖点种子"，放到 1.21.4 里可能根本找不到该群系，或位置偏移数千格。这类种子不能标注"1.21 通用"。

#### 第三层：小补丁版本（x.y.z → x.y.z+1）→ 一般兼容

Mojang 在纯 bugfix 补丁版本（如 1.21.3 → 1.21.4 中的热修复）间通常维持世界生成一致性。但仍建议以实测为准，不做无据的"向后兼容"保证。

### 2.2 版本兼容性标注规范

平台应提供两个版本字段：

- **精确测试版本**（必填）：投稿者实际测试的版本，如 `1.21.4`
- **已验证兼容范围**（选填）：如 `1.21.4 ~ 1.21.5`，需经实测，不可猜测填写

参考竞品 wisehosting 的做法，对每颗种子列出完整的已验证版本列表，以"版本徽章"形式展示。

### 2.3 基岩版版本号规范

基岩版使用三段式版本号（如 `1.21.60`），与 Java 版的语义完全不同，不可混用。平台应将两者的版本选择器完全分离。

## 3. 元信息字段设计

### 3.1 核心字段（全部必填）

```
seed_value        种子数值（字符串，保留前导负号）
edition           Java / 基岩版（单选）
tested_version    精确测试版本（如 1.21.4）
mod_env           原版 / 指定整合包（选后者必填整合包名+版本）
world_type        普通 / 大型群系 / 超平坦（影响地形，必须注明）
spawn_coords      出生点坐标（X, Z）
screenshots       出生点全景截图（至少 1 张，≤ 5 张）
tags              玩法标签（见 3.3，至少选 1 个）
title             种子标题（≤ 50 字）
description       种子描述（≤ 500 字，说明卖点、关键坐标等）
```

### 3.2 扩展字段（选填）

```
compatible_range  已验证兼容版本范围（如 1.21.4 ~ 1.21.5）
key_coords        关键坐标列表（结构名称 + X/Y/Z，可多条）
difficulty_note   难度说明（如"出生点无树，资源匮乏"）
uploader_note     投稿者补充（任意文本）
```

### 3.3 分类标签体系

标签分为三组，可多选，但每组至少选一个主标签：

**玩法向（主标签，必选其一）**
- 🏕️ 生存向 · ⚔️ 速通向 · 🏗️ 建筑向 · 💀 极限模式（Hardcore）· 🎯 挑战向

**景观/特性向（可多选）**
- 🌄 出生点奇观 · 🍄 稀有群系（蘑菇岛/苍白花园等）· 🏔️ 地形奇观（孤岛/环形山等）
- 🏘️ 近出生点村庄 · 🏰 近出生点要塞 · 🏛️ 古代城市 · ⚗️ 试炼室集群
- 💎 资源富集（钻石/矿脉）· 🌊 孤岛生存

**特殊标签**
- ✅ 已验证（由版主或社区验证） · 🔥 热门 · 🆕 新版本

## 4. 推荐项目架构

### 4.1 选型原则

- 面向 GitHub 开源，优先选择轻量、易自部署的技术栈
- 避免 MySQL（改用 SQLite）、Nginx（改用内置服务器或 Caddy）
- 前后端分离，方便社区贡献
- 图片存储使用本地文件系统或兼容 S3 的对象存储（如 MinIO、Cloudflare R2）

### 4.2 技术选型

| 层次 | 推荐方案 | 理由 |
|---|---|---|
| 前端框架 | **Vue 3 + Vite + TypeScript** | 你已熟悉，生态成熟 |
| UI 组件库 | **Naive UI** 或 **Element Plus** | 中文友好，文档完善 |
| 后端框架 | **FastAPI（Python）** | 轻量、自带文档、异步支持好 |
| 数据库 | **SQLite**（开发/小规模）→ **PostgreSQL**（可选升级路径） | SQLite 零配置，单文件，适合开源项目 |
| ORM | **SQLAlchemy 2.x + Alembic**（数据库迁移） | 成熟，迁移到 PG 无痛 |
| 图片存储 | 本地 `/uploads` 目录（开发）→ S3 兼容接口（生产） | 通过环境变量切换，零代码改动 |
| 认证 | **JWT**（HTTP-only Cookie） | 无状态，轻量 |
| 部署 | **单 Docker Compose 文件**，前端 build 后由 FastAPI 托管静态文件，或用 Caddy 反代 | 一键部署，无 Nginx |

### 4.4 数据库模型（核心表）

```sql
-- 用户表
users (id, username, email, password_hash, role, created_at)

-- 种子表（核心）
seeds (
  id, title, description,
  seed_value,              -- 种子数值（字符串存储，避免整数溢出）
  edition,                 -- 'java' | 'bedrock'
  tested_version,          -- 如 '1.21.4'
  compatible_range,        -- 如 '1.21.4~1.21.5'（可空）
  world_type,              -- 'normal' | 'large_biomes' | 'superflat'
  mod_env,                 -- 'vanilla' | 'modpack'
  modpack_name,            -- mod 整合包名（可空）
  modpack_version,         -- mod 整合包版本（可空）
  spawn_x, spawn_z,        -- 出生点坐标
  status,                  -- 'pending' | 'approved' | 'rejected'
  uploader_id,             -- FK → users.id
  view_count, like_count,
  created_at, updated_at
)

-- 截图表（一对多）
screenshots (id, seed_id, file_path, is_cover, sort_order)

-- 标签表（预置）
tags (id, name, category, icon)

-- 种子-标签关联
seed_tags (seed_id, tag_id)

-- 关键坐标表（一对多）
key_coords (id, seed_id, label, x, y, z)

-- 点赞表
likes (user_id, seed_id, created_at)
```

## 5. GUI 结构与功能设计

### 5.1 页面地图

```
/                首页（热门 + 最新 + 分类快捷入口）
/browse          浏览/搜索页（核心功能）
/seed/:id        种子详情页
/submit          投稿页（需登录）
/user/:username  用户主页
/admin           审核后台（管理员）
/login /register 认证页
```

### 5.2 各页面详细设计

---

#### 首页 `/`

**布局**：横幅 Banner → 快捷分类入口 → 热门种子卡片流 → 最新投稿

**功能**：
- Banner 展示当前最新版本（自动或手动维护）的推荐种子
- 分类快捷入口：生存 / 速通 / 建筑 / 极限 / 稀有群系，点击跳转到 `/browse?tag=xxx`
- 种子卡片：封面图 + 标题 + 版本徽章 + 标签 + 点赞数

---

#### 浏览/搜索页 `/browse`

**布局**：左侧筛选面板（桌面端）/ 顶部折叠筛选（移动端） + 右侧种子卡片瀑布流

**筛选器**（支持 URL 参数，方便分享和收藏）：

```
版本端：  [Java版] [基岩版]
版本号：  多选下拉（1.21.5 / 1.21.4 / 1.21.1 / ...）
玩法标签：[生存] [速通] [建筑] [极限] [挑战]
特性标签：[村庄] [要塞] [古代城市] [稀有群系] [孤岛] ...
Mod 环境：[仅原版] [含整合包]
排序：    [最热] [最新] [最多点赞]
```

**搜索框**：支持关键词搜索（匹配标题、描述）

**卡片设计**：
- 封面截图（16:9 裁切）
- 标题 + 简短描述（2 行截断）
- 版本徽章（如 `Java 1.21.4`）、玩法标签（最多 3 个）
- 点赞数 + 浏览数
- 投稿者头像 + 用户名

---

#### 种子详情页 `/seed/:id`

**区域划分**：

**左栏（主内容，约 65% 宽）**
- 截图轮播（支持全屏）
- 种子标题 + 描述
- 关键坐标列表（带复制按钮，格式：`结构名 X: 123 Y: 64 Z: -456`）
- 评论区（Markdown 支持，可 @ 用户）

**右栏（种子信息卡片，约 35% 宽）**
- **种子数值**（大字体，一键复制按钮，点击后显示"已复制！"）
- 版本信息：版本端、精确版本、兼容范围
- 世界类型、Mod 环境
- 出生点坐标
- 标签组
- 投稿者信息 + 投稿时间
- 点赞按钮（已登录才可操作）
- 举报按钮

---

#### 投稿页 `/submit`

**表单分步设计（3 步，减轻填写压力）**：

**第 1 步：基本信息**
- 种子数值（输入框，自动检测是否为合法数值，支持纯字符种子）
- 版本端（Java / 基岩版，单选，切换后版本号选择器联动刷新）
- 精确测试版本（下拉，按端分别维护版本列表）
- 世界类型（单选）
- Mod 环境（单选，选"整合包"后展开整合包名/版本输入框）

**第 2 步：截图与坐标**
- 截图上传（拖拽 / 点击，最多 5 张，第一张自动设为封面，可拖拽排序）
- 出生点坐标（X / Z，必填）
- 关键坐标（动态列表，可添加多条，每条包含：名称 + X/Y/Z）

**第 3 步：标题与标签**
- 种子标题（≤ 50 字，带字数提示）
- 种子描述（富文本 / Markdown，≤ 500 字）
- 标签选择（玩法标签必选 1 个，特性标签可多选）
- 兼容版本范围（选填，说明"经实测此种子在以下版本表现一致"）
- 提交前预览

**表单验证要点**：
- 种子数值唯一性检查（同版本+同版本端不允许重复投稿）
- 截图文件大小限制（单张 ≤ 5 MB，支持 JPG/PNG/WebP）
- 坐标值格式校验（整数）

---

#### 审核后台 `/admin`

- 待审种子列表（默认视图）
- 快速预览（不跳转详情页，侧滑抽屉展示）
- 审核操作：通过 / 拒绝（拒绝需填写原因，自动发送通知给投稿者）
- 版本管理：维护各端的版本号列表（新版本发布后在此添加，自动同步到投稿表单的下拉选项）
- 用户管理：封禁 / 解封

---

### 5.3 交互细节

- **种子数值复制**：详情页核心交互，按钮需足够大，Mobile 端尤其重要
- **版本徽章**：以不同颜色区分 Java（绿）/ 基岩（橙），避免混淆
- **兼容性提示**：若种子的精确测试版本非当前最新版，在详情页顶部显示提示横幅（"此种子在 1.21.4 下测试，当前最新版为 1.21.5，请注意兼容性"）
- **空状态设计**：搜索无结果时展示友好提示 + 引导投稿

## 6. 其他重要议题

### 6.1 内容审核机制

开放投稿意味着必须有审核流程，建议采用**先审后发**模式：

- 新用户投稿进入 `pending` 队列，等待人工审核
- 积累一定信誉值（如 5 个被通过的投稿）后升级为"信任用户"，享有**快速通道**（审核优先级更高，或直接发布）
- 审核标准应公开：截图是否与描述匹配、版本信息是否填写准确、是否存在重复投稿

### 6.2 版本维护的持续运营成本

这是容易被忽视的隐性成本。Mojang 每隔数月就发布一次版本，每次版本更新后：

1. 需要手动在后台添加新版本号（低成本，几分钟）
2. 需要评估新版本是否改变了世界生成（需要有人实测，中等成本）
3. 如果世界生成有重大变化，需要在站内通知社区（低成本）

建议在项目 README 中明确说明版本维护规则，或建立一个"版本更新响应流程"文档，吸引社区协作维护。

### 6.3 种子去重与防刷

同一颗种子可以被无限投稿，需要策略防止：

- **精确去重**：seed_value + edition + tested_version 三者相同视为重复，在投稿时前端实时检测并提示"该种子已有投稿，可查看现有条目"
- **相似版本去重提示**：如 1.21.4 的某种子已存在，投稿 1.21.5 相同种子时提示"此种子已有 1.21.4 版本条目，请确认两版本表现是否一致"
- **刷赞防护**：同一用户对同一种子只能点赞一次，服务端校验

### 6.4 SEO 与分享优化

种子详情页的 URL 应该对搜索引擎友好，且分享到社交平台时有完整的 Open Graph 预览：

```
URL 示例：/seed/12345-古代城市出生点-1.21.4-java
OG 图片：自动取种子封面截图
OG 描述：自动截取种子描述前 120 字
```

### 6.5 关于竞品差异化的再确认

竞品（seedb.co.uk、MCSeedShare）的核心缺失：

- **无中文界面**：国内玩家使用体验差
- **无玩法向分类**：多以地形/群系为维度，缺乏"速通友好"、"建筑胜地"等玩法导向标签
- **无 Mod 整合包种子**：整合包社群的需求完全未被满足
- **无版本兼容性徽章**：不直观展示跨版本可用情况

这些正是本平台的差异化机会，也应是 MVP 阶段优先确保做好的功能。

### 6.6 MVP 范围建议

第一个可发布版本应包含且仅包含以下功能，避免过度设计：

**必须有**：
- 种子列表页（含基础筛选：版本端、版本号、玩法标签）
- 种子详情页（含截图、坐标、一键复制种子值）
- 投稿表单（完整字段，人工审核）
- 简单的用户注册/登录（仅用于投稿和点赞）
- 管理员审核后台

**可以后期迭代**：
- 评论系统
- 用户主页 / 关注系统
- 信誉系统
- 整合包种子专区
- 种子地图可视化（接入 Chunkbase API 或开源的 cubiomes-wasm）
- RSS / API 开放

### 6.7 种子地图可视化（未来功能）

竞品中最吸引人的功能之一是地图预览（如 Chunkbase、MineNest 的 Seed Map），可以在浏览器中直接查看群系分布。实现方案：

- **开源方案**：[cubiomes](https://github.com/Cubitect/cubiomes) 是一个 C 语言种子分析库，有 WebAssembly 编译版本，可在浏览器端直接运行，无需服务端渲染
- **接入成本**：中等，需要一定的 WASM 集成工作，建议作为 v2 功能
